# Writing Macro Automation Studio Python scripts

A macro is a Python program that drives an Android emulator (or other device) via
the `mas` SDK. The script reads its arguments from a generated `script_args.py`,
performs device automation (click, swipe, find image, OCR, app control, persistent
storage), and optionally updates a runtime dashboard via `mas.ui.*`.

## Standard project layout

```
my-macro/
‚îú‚îÄ‚îÄ my-macro.uibproj          # UIBuilder argument-form source
‚îú‚îÄ‚îÄ my-macro.uibrt            # UIBuilder runtime-dashboard source (optional)
‚îú‚îÄ‚îÄ ui.xml                    # generated by UIBuilder on save (do not edit)
‚îú‚îÄ‚îÄ requirements.txt          # third-party Python deps (mas SDK is pre-installed)
‚îî‚îÄ‚îÄ src/
    ‚îú‚îÄ‚îÄ app.py                # entry point ‚Äî this is what we write
    ‚îî‚îÄ‚îÄ script_args.py        # generated by UIBuilder from .uibproj (do not edit)
```

The host app launches `src/app.py` with the CLI flags built from the runner dialog.
`src/app.py` imports `from src.script_args import args` to read those flags as a
typed namespace, then runs the macro logic.

See [macro-project-structure.md](references/macro-project-structure.md) for details.

## Workflow

1. **Clarify the goal.** What does the macro actually do? Open an app, find an
   image, click it, repeat? Read OCR off a screen and store results? Drive a UI
   form? Don't guess ‚Äî ask if needed.
2. **Identify inputs.** What does the user need to provide before each run? Those
   become argument-form widgets (use the `generating-args-form` skill when the
   form doesn't exist yet) and get read from `args.<tab>.<key>`.
3. **Identify live signals.** Anything the user wants to watch during execution ‚Äî
   progress, status, log lines, charts ‚Äî becomes runtime widgets driven via
   `mas.ui.*` (use `generating-runtime-dashboard` for the .uibrt). Skip if
   there's no need for a dashboard.
4. **Pick SDK calls** for each step. See `references/mas-sdk-overview.md` for the
   namespace map; per-namespace details in the references below.
5. **Write `src/app.py`.** Skeleton: imports ‚Üí top-level setup (open app, register
   images) ‚Üí main loop ‚Üí cleanup. Wrap iterations in try/except using the
   patterns in `references/error-handling-patterns.md`.
6. **Show the user the matching `mas.images({...})` declaration** if the macro uses
   `find_object` ‚Äî those integer IDs come from the macro's Image Library and the
   user supplies them, you don't invent them.

## Output

Default to writing the script to `src/app.py` if the user has a project path, or
print the file in a fenced Python block. Always:

- Use `from src.script_args import args` for CLI args (don't roll your own argparse).
- Use `mas.images({"name": int_id, ‚Ä¶})` rather than passing raw int IDs around.
- Use `mas.wait_for_object(...)` rather than busy-loop polling.
- Wrap UI-updating bursts in `with mas.ui.batch():` when many updates happen close
  together.
- Catch `mas.DeviceNotConnectedError`, `mas.ImageNotFoundError`, `mas.TimeoutError`
  at the loop boundary, not silently inside it.
